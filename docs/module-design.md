# 模組設計書 (Module Design)

## 一、功能模組切割（Feature Modules）

以使用者可感知的功能為切割單位。

---

### F1：股票資料查詢

**目的**：使用者輸入股票代號，系統取得該股票的歷史交易資料

| 項目 | 說明 |
|---|---|
| 輸入 | 台股股票代號（如 `2330`） |
| 輸出 | 最近 30 個交易日的每日資料（日期、開高低收、成交量） |
| 具體行為 | 1. 驗證代號是否有效 |
|  | 2. 透過 Data Fetcher 取得歷史資料（開發期用 seed data，上線切換為 TWSE API） |
|  | 3. 從中截取最近 30 個交易日 |
|  | 4. 補充當日本益比、殖利率等（TWSE OpenAPI / 本地 JSON） |
| 異常處理 | 代號不存在 → 提示「查無此股票」 |
|  | API 無回應 → 提示「資料來源暫時無法連線」 |
|  | 資料筆數不足 30 天 → 回傳實際筆數並帶警告 |

---

### F2：技術指標計算

**目的**：根據歷史收盤價計算 MA5 / MA20

| 項目 | 說明 |
|---|---|
| 輸入 | F1 取得的每日收盤價序列 |
| 輸出 | 每個交易日對應的 MA5、MA20 數值 |
| 具體行為 | 1. MA5 = 最近 5 日收盤價的簡單平均 |
|  | 2. MA20 = 最近 20 日收盤價的簡單平均 |
|  | 3. 前 4 筆無 MA5、前 19 筆無 MA20（標記為 null） |
| 異常處理 | 資料不足 5 筆 → 無法計算 MA5，帶警告 |
|  | 資料不足 20 筆 → 無法計算 MA20，帶警告 |
|  | 收盤價為空值 → 跳過該筆，帶警告 |

---

### F3：股價走勢圖表

**目的**：以圖表呈現股價走勢與均線

| 項目 | 說明 |
|---|---|
| 輸入 | F1 的每日資料 + F2 的 MA5/MA20 |
| 輸出 | 互動式折線圖 |
| 具體行為 | 1. X 軸：交易日期，Y 軸：價格 |
|  | 2. 收盤價折線（主線） |
|  | 3. MA5 曲線（不同顏色） |
|  | 4. MA20 曲線（不同顏色） |
|  | 5. 滑鼠懸停顯示當日數值 |
| 異常處理 | MA 值為 null 的日期不繪製該段曲線 |
|  | 資料不足時圖表仍顯示可用部分，搭配警告文字 |

---

### F4：AI 分析報告

**目的**：根據實際數據產生繁體中文的客觀分析文字

| 項目 | 說明 |
|---|---|
| 輸入 | F1 的每日資料 + F2 的 MA5/MA20 |
| 輸出 | 一段繁體中文分析文字 |
| 具體行為 | 1. **事實提取**（規則引擎）：從結構化數據中提取具體事實 |
|  | 2. 事實不足 2 項 → 拒絕輸出 |
|  | 3. **文字生成**：模板模式（預設）或 AI 改寫模式（有 API Key 時） |
|  | 4. 前端標註「AI 生成內容」 |
| 約束條件 | 僅根據系統中實際顯示的數據進行描述 |
|  | 至少引用 2 項具體數據事實（均線關係、價格變化等） |
|  | 不可提供投資建議、預測或主觀推薦 |
|  | 不可使用無法由資料直接支持的推論 |
| 異常處理 | 事實不足 2 項 → 不產生分析，顯示「依據不足，無法產生分析」 |
|  | AI API 失敗 → 降級為模板模式（不影響功能） |

---

### F5：資料可信度檢查

**目的**：讓使用者了解資料是否完整、指標是否可信

| 項目 | 說明 |
|---|---|
| 輸入 | F1 的原始資料狀態 + F2 的計算結果 |
| 輸出 | 一組 warnings 列表，前端以醒目方式顯示 |
| 檢查項目 | 1. 資料筆數是否滿 30 個交易日 |
|  | 2. 是否有缺漏欄位（收盤價為空等） |
|  | 3. MA5 / MA20 是否可被正確計算 |
|  | 4. AI 分析的數據依據是否充足 |
| 原則 | 有疑慮就提示，不隱藏問題，不輸出可能誤導的結果 |

---

### 功能模組關係

```
使用者輸入代號
     │
     ▼
   [F1 股票資料查詢] ──取得原始資料──▶ [F5 資料可信度檢查]
     │                                      │
     ▼                                      │ 產生 warnings
   [F2 技術指標計算]                         │
     │                                      │
     ├──────────────────────────────────────┘
     │
     ▼
   [F3 股價走勢圖表] ← 資料 + MA + warnings
   [F4 AI 分析報告]  ← 資料 + MA + warnings
```

## 二、架構層級切割（Architecture Layers）

以技術職責為切割單位。

---

### L1：資料取得層 (Data Fetcher)

**職責**：封裝所有資料來源的存取邏輯，透過統一介面提供資料，使上層不需關心資料來自哪裡

| 具體工作 | 說明 |
|---|---|
| Seed Data（開發期） | 讀取本地 seed JSON 檔案，立即回傳，無需網路 |
| TWSE 傳統 API（正式） | `GET /exchangeReport/STOCK_DAY?response=json&date={YYYYMMDD}&stockNo={code}`，查 2~3 個月合併 |
| TWSE OpenAPI（補充） | `GET /v1/exchangeReport/BWIBBU_ALL` 等，取當日本益比、殖利率 |
| 本地 JSON（補充） | 讀取 `BWIBBU_ALL.json` 等，作為 TWSE OpenAPI 的離線備援 |
| 來源切換機制 | 透過設定（環境變數或 config）切換 seed / TWSE，上層程式碼不需修改 |
| 統一錯誤處理 | API 逾時、無資料、格式異常等統一回傳錯誤物件 |

---

### L2：資料處理層 (Data Processor)

**職責**：清洗、轉換、計算，產出結構化結果

| 具體工作 | 說明 |
|---|---|
| 資料清洗 | 移除逗號、空值處理、字串轉數字、民國曆轉西元 |
| 截取 30 個交易日 | 從原始資料中取出最近 30 筆 |
| 計算 MA5 / MA20 | 簡單移動平均 |
| 資料完整性檢查 | 產生 warnings 列表 |

---

### L3：AI 分析層 (AI Analyzer)

**職責**：從結構化數據中提取事實，生成符合規範的繁中分析文字

#### 設計決策：兩層架構

因 AI API 資源不保證隨時可用，採用「事實提取 + 文字生成」兩層分離設計，確保系統永遠能產出分析結果。

**第一層：事實提取（規則引擎，永遠執行）**

從 L2 產出的結構化資料中，提取可被數據直接佐證的事實：

| 提取項目 | 範例 |
|---|---|
| 期間價格變化 | 「收盤價從 1,880 元下跌至 1,810 元，跌幅 3.72%」 |
| MA5 vs MA20 關係 | 「MA5（1,805 元）低於 MA20（1,844 元），短期均線位於長期均線下方」 |
| 期間最高/最低價 | 「期間最高價 2,015 元（02/25），最低價 1,760 元（03/31）」 |
| MA 交叉事件 | 「MA5 於 03/20 向下穿越 MA20」（若有發生） |
| 近期趨勢方向 | 「近 5 個交易日收盤價呈下降趨勢」 |

每項事實皆可追溯至具體數據，符合「不可使用無法由資料直接支持的推論」之規範。

**拒絕輸出判斷**：提取的事實不足 2 項（例如資料太少）→ 回傳「依據不足，無法產生分析」，不硬湊內容。

**第二層：文字生成（兩種模式）**

| 模式 | 觸發條件 | 做法 |
|---|---|---|
| 模板模式（預設） | 無 `AI_API_KEY` 環境變數 | 用預寫的繁中模板將事實列表串接為段落 |
| AI 改寫模式 | 有 `AI_API_KEY` 且呼叫成功 | 將已提取的事實列表交給 AI 改寫為更流暢的段落 |

**AI 改寫模式的關鍵約束**：
- AI 收到的是**已提取的事實列表**，不是原始數據
- Prompt 明確限制：「請用以下事實改寫為一段流暢的繁體中文段落，不可新增、推論或省略任何事實，不可提供投資建議或預測」
- AI 的角色是**文字潤飾**，不是數據分析
- 呼叫失敗 → 降級為模板模式，不影響功能

#### 完整流程

```
結構化資料（data + MA5 + MA20 + warnings）
        │
        ▼
  第一層：事實提取（規則引擎）
  → facts = [
      "收盤價從 1,880 元下跌至 1,810 元，跌幅 3.72%",
      "MA5（1,805 元）低於 MA20（1,844 元）",
      "期間最高價 2,015 元（02/25），最低價 1,760 元（03/31）",
    ]
        │
        ├─ facts 不足 2 項 → 回傳「依據不足，無法產生分析」
        │
        ▼
  第二層：文字生成
        │
        ├─ 無 AI_API_KEY → 模板串接 → 回傳分析文字
        │
        └─ 有 AI_API_KEY → AI 改寫 facts
                             ├─ 成功 → 回傳 AI 版本
                             └─ 失敗 → 降級為模板版本
```

#### 為何此設計符合需求規範

| 規範 | 如何保證 |
|---|---|
| 僅根據實際顯示的數據描述 | 事實提取層只從 L2 結構化資料中取值 |
| 至少引用 2 項具體數據事實 | 事實不足 2 項時拒絕輸出 |
| 不可投資建議、預測、主觀推薦 | 規則引擎不具備此能力；AI 模式的 prompt 明確禁止 |
| 不可使用無法由資料支持的推論 | AI 只能改寫已提取事實，無法接觸原始數據自行推論 |
| 資料不足時不輸出誤導結果 | 事實提取前先檢查 warnings，依據不足直接拒絕 |

---

### L4：後端 API 層 (Backend API)

**職責**：對外提供 HTTP 介面，串接 L1 → L2 → L3

| 具體工作 | 說明 |
|---|---|
| 端點定義 | `GET /api/stock/{code}` |
| 流程編排 | 依序呼叫各層，組裝完整回應 |
| 統一回應格式 | 成功 / 失敗皆有一致的 JSON 結構 |

**回應結構**：

```json
{
  "stock": { "code": "2330", "name": "台積電" },
  "data": [
    {
      "date": "2026-03-02",
      "open": 1840,
      "high": 1855,
      "low": 1830,
      "close": 1850,
      "volume": 25000000,
      "ma5": 1842,
      "ma20": 1810
    }
  ],
  "analysis": "近 30 個交易日收盤價從 ... 至 ...",
  "warnings": ["MA20 前 19 筆無法計算"],
  "extra": { "pe_ratio": "20.5", "dividend_yield": "2.1", "pb_ratio": "4.5" }
}
```

---

### L5：前端展示層 (Frontend)

**職責**：使用者介面，處理互動與呈現

| 具體工作 | 說明 |
|---|---|
| 輸入介面 | 股票代號輸入框 + 查詢按鈕 |
| 圖表渲染 | 收盤價折線 + MA5/MA20 曲線 |
| 分析文字顯示 | AI 分析區塊，標註「AI 生成內容」 |
| 警告提示 | 醒目顯示 warnings |
| 錯誤狀態 | 代號無效、API 失敗等使用者提示 |

---

### 架構層級關係

```
[L5 Frontend]
     │
     │  HTTP Request
     ▼
[L4 Backend API]
     │
     ├──▶ [L1 Data Fetcher] ──▶ Seed Data / TWSE API（可切換）
     │          │
     │          ▼ 原始資料
     │
     ├──▶ [L2 Data Processor] ──▶ 清洗 + MA5/MA20 + warnings
     │          │
     │          ▼ 結構化資料
     │
     └──▶ [L3 AI Analyzer] ──▶ 分析文字
              │
              ▼
        組裝回應 → 回傳 L5
```

| 功能模組 | 涉及的架構層級 |

| F1 股票資料查詢 | L1 Data Fetcher + L2 Data Processor（清洗部分） |

## 四、Seed Data 策略

### 4.1 目的

開發期間以本地靜態資料替代 TWSE API，優先開發功能邏輯，資料來源隨時可切換。

### 4.2 Seed Data 來源

從現有本地 JSON + TWSE API 實際回傳格式，產生模擬的 30 個交易日資料。

**seed 檔案規劃**：

```
twstock-30d-viewer/seed/
  ├── stock_2330.json    ← 台積電 30 個交易日（主要開發用）
  ├── stock_0050.json    ← 元大台灣50（測試 ETF）
  └── stock_invalid.json ← 模擬異常情境（資料不足、欄位缺漏等）
```

### 4.3 Seed Data 格式

與 TWSE 傳統 API 回傳格式一致，確保切換時無需修改處理邏輯：

```json
{
  "stat": "OK",
  "title": "115年04月 2330 台積電各日成交資訊",
  "fields": ["日期", "成交股數", "成交金額", "開盤價", "最高價", "最低價", "收盤價", "漲跌價差", "成交筆數"],
  "data": [
    ["115/02/03", "25,456,789", "47,000,000,000", "1,850.00", "1,860.00", "1,840.00", "1,855.00", "+10.00", "30,000"],
    ["115/02/04", "..."],
    "... 共 30+ 筆"
  ],
  "total": 30
}
```

### 4.4 異常情境 Seed Data

`stock_invalid.json` 包含以下測試情境：

| 情境 | 模擬方式 |
|---|---|
| 資料不足 | data 僅 10 筆（不足 20 筆，MA20 無法計算） |
| 欄位缺漏 | 部分筆的收盤價為空字串 `""` |
| 代號不存在 | `stat: "很抱歉，沒有符合條件的資料!"`, 無 data 欄位 |

### 4.5 來源切換機制

```python
# config.py 或環境變數
DATA_SOURCE = "seed"  # "seed" | "twse"

# data_fetcher.py
class DataFetcher:
    def fetch(self, stock_no: str) -> dict:
        if DATA_SOURCE == "seed":
            return self._from_seed(stock_no)
        else:
            return self._from_twse(stock_no)

    def _from_seed(self, stock_no: str) -> dict:
        """讀取 seed/{stock_no}.json"""
        ...

    def _from_twse(self, stock_no: str) -> dict:
        """呼叫 TWSE 傳統 API，合併 2~3 個月資料"""
        ...
```

**切換方式**：只需改 `DATA_SOURCE` 設定值，其餘程式碼（L2 處理層、L3 分析層、L4 API 層、L5 前端）完全不變。

### 4.6 開發順序建議

```
Phase 1 ── 用 seed data 開發
  ├─ 建立 seed 資料（含正常 + 異常情境）
  ├─ L2 Data Processor（清洗 + MA 計算 + 完整性檢查）
  ├─ L4 Backend API（端點 + 回應格式）
  ├─ L5 Frontend（圖表 + 輸入 + 警告）
  └─ L3 AI Analyzer（prompt + API 呼叫）

Phase 2 ── 切換為 TWSE API
  ├─ 實作 _from_twse()（多月查詢 + 合併 + 頻率控制）
  ├─ DATA_SOURCE 改為 "twse"
  └─ 端對端驗證
```
