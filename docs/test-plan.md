# 測試計畫（規格驅動）

本文件所有測試案例皆從需求規格書（origin-spec.md）、模組設計書（module-design.md）、風險處理文件（risk-handling.md）推導而來，不以現有程式碼實作為依據。

測試原則：**驗證系統「該做什麼」與「不該做什麼」，而非「程式碼做了什麼」。**

---

## 測試架構總覽

```
twstock-30d-viewer/
├── backend/tests/       ← 後端單元 + 整合測試（pytest）
├── frontend/tests/      ← 前端單元 + 元件測試（Vitest）
└── tests/               ← 端對端測試（Playwright，前後端聯合）
```

| 層級 | 位置 | 測什麼 | 怎麼跑 |
|------|------|--------|--------|
| 後端單元測試 | `backend/tests/` | 單一函式的輸入輸出（清洗、MA 計算、事實提取） | `cd backend && pytest` |
| 後端整合測試 | `backend/tests/` | 打 API endpoint，驗完整回應結構與內容 | `cd backend && pytest` |
| 前端單元測試 | `frontend/tests/` | API 呼叫封裝、資料轉換邏輯 | `cd frontend && npx vitest run` |
| 前端元件測試 | `frontend/tests/` | render 元件，驗畫面是否正確呈現資料與警告 | `cd frontend && npx vitest run` |
| 端對端測試 | `tests/` | 啟動瀏覽器，模擬使用者操作完整流程 | `cd tests && npx playwright test` |

## B-1. 資料清洗與轉換

> 檔案：`test_data_processor.py`
> 依據：origin-spec.md 第六章（TWSE 原始資料格式）

### B-1.1 民國曆轉換

- [x] `115/04/01` → `2026-04-01`
- [x] `115/01/02` → `2026-01-02`（跨年）
- [x] `114/12/31` → `2025-12-31`（前一年）
- [x] `100/01/01` → `2011-01-01`（民國 100 年）

### B-1.2 數值格式清洗

- [x] `"1,870.00"` → `1870.0`（去除逗號）
- [x] `"+15.00"` → `15.0`（正號）
- [x] `"-15.00"` → `-15.0`（負號）
- [x] `"47,389,975"` → `47389975`（大數字逗號）
- [x] `""` → 被識別為空值（非轉為 0）
- [x] `"--"` → 被識別為無效值

### B-1.3 整筆異常處理

- [x] 欄位數量不足 9 個的資料列 → 整筆略過
- [x] 日期格式無法解析的資料列 → 整筆略過
- [x] 某筆收盤價為空值 → 該筆被跳過，不參與後續計算
- [x] 某筆收盤價為 0 → 該筆被跳過（0 元不是合理股價）
- [x] 日期亂序的原始資料 → 自動排序後，結果與手動排序一致

## B-2. 技術指標計算

> 檔案：`test_data_processor.py`
> 依據：origin-spec.md 2.1、module-design.md F2

### B-2.1 MA5 正確性

- [x] 任取一天的 MA5，手動以該日往前 5 日收盤價平均驗證，結果一致
- [x] 前 4 筆資料的 MA5 為 null（不足 5 日無法計算）
- [x] 第 5 筆開始 MA5 皆有值
- [x] MA5 計算不受開盤價、最高價、最低價影響（僅使用收盤價）

### B-2.2 MA20 正確性

- [x] 任取一天的 MA20，手動以該日往前 20 日收盤價平均驗證，結果一致
- [x] 前 19 筆資料的 MA20 為 null
- [x] 第 20 筆開始 MA20 皆有值

### B-2.3 邊界情境

- [x] 資料恰好 5 筆 → MA5 僅最後 1 筆有值，MA20 全部為 null
- [x] 資料恰好 20 筆 → MA20 僅最後 1 筆有值
- [x] 資料恰好 30 筆 → MA5 有 26 個值，MA20 有 11 個值
- [x] 資料 0 筆 → 不計算，不回傳誤導性數字
- [x] 資料 1 筆 → MA5 = null, MA20 = null
- [x] 連續多筆收盤價異常 → 跳過後剩餘資料仍能正確計算 MA

## B-3. AI 分析引擎

> 檔案：`test_ai_analyzer.py`
> 依據：origin-spec.md 2.3、module-design.md F4 / L3

### B-3.1 分析內容規範

- [x] 分析文字為繁體中文
- [x] 至少引用 2 項具體數據事實（可從文字中找到具體數字）
- [x] **不包含**投資建議用語（如「建議買入」「適合進場」「值得關注」）
- [x] **不包含**預測用語（如「預計將」「未來可能」「看好後市」）
- [x] **不包含**主觀推薦（如「表現優異」「值得期待」）
- [x] 文字中引用的每個數字，皆可在輸入的 data 中找到對應來源

### B-3.2 分析依據充足性

- [x] 正常 30 筆資料 → 產生分析文字，非空非 null
- [x] 事實提取不足 2 項 → 回傳「依據不足，無法產生分析」，不硬湊內容
- [x] 資料 0 筆 → 不產生分析

### B-3.3 分析可追溯性

- [x] 分析提及「收盤價從 X 元至 Y 元」→ X 確實是期間第一天收盤價，Y 確實是最後一天
- [x] 分析提及「MA5 為 X 元」→ X 與輸入資料中對應日期的 ma5 值一致
- [x] 分析提及「最高價 X 元（MM/DD）」→ 該日期的最高價確實為 X
- [x] 分析提及均線交叉 → 輸入資料中確實存在 MA5 與 MA20 的交叉點

### B-3.4 降級機制

- [x] 無 AI API Key → 仍能產生分析文字（模板模式）
- [x] AI API 呼叫失敗 → 降級為模板模式，回傳值結構不變
- [x] AI API 呼叫失敗時，回傳內容不包含 API 錯誤細節（如 API key、endpoint）

## B-4. API 回應契約（整合測試）

> 檔案：`test_api.py`
> 依據：module-design.md L4（回應結構定義）

### B-4.1 成功回應結構

- [x] 回應包含 `stock` 物件，含 `code`（字串）與 `name`（字串）
- [x] 回應包含 `data` 陣列，每筆含 `date`, `open`, `high`, `low`, `close`, `volume`, `ma5`, `ma20`
- [x] `data[].date` 格式為 `YYYY-MM-DD`
- [x] `data[].close` 為數字（非字串）
- [x] `data[].ma5` 為數字或 null
- [x] `data[].ma20` 為數字或 null
- [x] 回應包含 `analysis`（字串或 null）
- [x] 回應包含 `warnings`（字串陣列，可為空陣列）

### B-4.2 錯誤回應結構

- [x] 代號無效時，回應包含 `error` 欄位，值為使用者可讀的中文訊息
- [x] 錯誤回應不包含程式內部資訊（如 file path、stack trace）

### B-4.3 一致性與完整性

- [x] 同一代號連續查詢兩次（seed 模式），回傳結果完全一致（冪等性）
- [x] `data` 陣列長度與 `warnings` 內容一致（如 data 有 22 筆，warnings 應提及不足 30 日）
- [x] 日期嚴格遞增排列（第 N 筆日期 < 第 N+1 筆）
- [x] 回傳的 30 筆皆為實際交易日（不含週末與假日）

### B-4.4 無效輸入

- [x] 輸入不存在的代號(如 `9999`) → 回傳明確的錯誤訊息
- [x] 輸入空字串 → 回傳錯誤，不回傳空資料
- [x] 輸入非數字字串(如 `abcd`) → 回傳錯誤
- [x] 輸入帶有特殊字元的字串(如 `23;30`) → 回傳錯誤，不觸發非預期行為

### B-4.5 資料來源異常（需 mock 網路）

- [x] API 連線逾時 → 回傳使用者可理解的錯誤訊息（非技術性 stack trace）
- [x] API 回傳非 JSON 格式 → 不 crash，回傳錯誤訊息
- [x] API 回傳 `stat` 非 "OK" → 回傳錯誤訊息
- [x] 第一個月查詢失敗 → 不繼續查後續月份，直接告知無法連線
- [x] 第二或第三個月查詢失敗 → 用已取得的資料繼續，但帶警告

### B-4.6 資料可信度（warnings 驗證）

- [x] 資料不足 30 個交易日 → warnings 包含筆數不足的訊息
- [x] 資料不足 5 筆 → warnings 包含無法計算 MA5 的訊息
- [x] 資料不足 20 筆 → warnings 包含無法計算 MA20 的訊息
- [x] 某筆收盤價為空 → warnings 包含該筆被略過的訊息
- [x] 日期排序異常 → warnings 包含已自動重新排序的訊息
- [x] 無任何異常時 → warnings 為空陣列

# 測試資料規劃

三份 fixtures 各自獨立，互不依賴，也不與開發用的 `seed/` 混用。

```
seed/                          ← 開發用，給 data_fetcher seed 模式讀取，不用於任何測試
backend/tests/fixtures/        ← 後端測試專用，輸入是 TWSE 原始格式，附預期答案
frontend/tests/fixtures/       ← 前端測試專用，輸入是後端 API 回應格式
tests/fixtures/                ← 端對端測試專用，設定後端使用的 seed 資料
```

## 後端 fixtures（backend/tests/fixtures/）

輸入格式：TWSE 傳統 API 的原始 JSON 格式（含民國曆、逗號數字）。
有明確的預期答案，不隨開發調整。

| 檔案 | 用途 | 關鍵特徵 |
|------|------|---------|
| `normal_30.json` | 驗證正常路徑 | 30 筆完整資料，所有欄位有值，日期連續遞增 |
| `with_ma_answers.json` | 驗證 MA 正確性 | 附帶手動計算的 MA5/MA20 預期值 |
| `insufficient_5.json` | 驗證不足 5 筆 | MA5 僅最後 1 筆有值，MA20 全為 null |
| `insufficient_22.json` | 驗證不足 30 日 | MA20 僅最後幾筆有值，應有筆數不足警告 |
| `disordered.json` | 驗證排序修正 | 日期非遞增，排序後應與 normal 結果一致 |
| `zero_close.json` | 驗證 0 值防護 | 某筆收盤價為 `"0.00"`，應被跳過 |
| `empty_fields.json` | 驗證空值處理 | 收盤價為 `""`、欄位數不足等混合情境 |
| `invalid_response.json` | 驗證錯誤處理 | stat 非 OK，無 data 欄位 |

## 前端 fixtures（frontend/tests/fixtures/）

輸入格式：後端 `GET /api/stock/{code}` 的回應 JSON（已清洗、已計算 MA）。
模擬後端 API 回應，不依賴後端啟動。

| 檔案 | 用途 | 關鍵特徵 |
|------|------|---------|
| `success.json` | 驗證正常渲染 | 完整的成功回應，含 stock / data / analysis / warnings |
| `with_warnings.json` | 驗證警告顯示 | warnings 陣列含多筆警告 |
| `error.json` | 驗證錯誤處理 | 含 error 欄位的回應 |
| `null_analysis.json` | 驗證 analysis 為 null | analysis 為 null，其餘正常 |

## 端對端 fixtures（tests/fixtures/）

端對端測試需要前後端皆啟動。後端切換為 seed 模式，讀取此目錄下的資料。
這些資料與 `seed/` 內容可能相似，但由測試獨立維護，確保測試結果不受開發調整影響。

| 檔案 | 用途 | 關鍵特徵 |
|------|------|---------|
| `stock_2330.json` | 正常查詢流程 | 30+ 筆完整資料，E-1 全部案例依賴此資料 |
| `stock_few.json` | 資料不足流程 | 不足 30 筆，觸發警告與 MA 不足情境 |
| `stock_invalid.json` | 錯誤處理流程 | stat 非 OK，驗證錯誤畫面 |
