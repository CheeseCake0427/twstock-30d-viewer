## 1. 連線失敗

**如何偵測**：`data_fetcher.py` 的 `_fetch_one_month()` 用 try/except 包住 `requests.get()`，捕捉所有 exception（含 timeout、SSL、網路中斷），失敗回傳 `None`。上層 `_from_twse()` 檢查 `result is None`。

**是否繼續計算**：
- 第一個月就失敗 → 不繼續，直接回傳 `stat: "資料來源暫時無法連線，請稍後再試"`
- 第二或第三個月才失敗 → 用已取得的資料繼續處理，筆數可能不足 30，會由 processor 產生 warning

**如何提示使用者**：前端收到 `error` 欄位 → 顯示紅色錯誤區塊，內容為後端回傳的訊息。

**決策原則**：有部分資料就盡量顯示，但明確告知不完整；完全沒資料就直接擋住，不輸出空圖表。

**實作狀態**：已實作。

---

## 2. 回傳資料筆數不足

**如何偵測**：`data_processor.py` 清洗後檢查筆數：`total < 30`（不足 30 日）、`total < 5`（無法算 MA5）、`total < 20`（無法算 MA20）。

**是否繼續計算**：
- 繼續。有多少筆就算多少，MA5/MA20 在筆數不足的位置回傳 `null`
- 0 筆 → 不計算，回傳空 data + warning「無有效交易日資料」

**如何提示使用者**：warnings 陣列加入具體訊息：
- 「資料僅 X 個交易日，不足 30 日」
- 「資料不足 5 筆，無法計算 MA5」
- 「資料不足 20 筆，無法計算 MA20」

前端以黃色警告區塊顯示。AI 分析層若事實不足 2 項，回傳「依據不足，無法產生分析」。

**決策原則**：不隱藏資料缺口，有多少就顯示多少，但明確標註哪些指標無法計算，不輸出可能誤導的結果。

**實作狀態**：已實作。

---

## 3. 日期排序錯誤

**如何偵測**：`data_processor.py` 清洗完成後、截取 30 筆之前，逐筆比較 `cleaned[i]["date"]` 與 `cleaned[i-1]["date"]`（ISO 格式字串比較），檢查是否嚴格遞增。

**是否繼續計算**：繼續。發現亂序時自動以日期排序（`cleaned.sort(key=lambda d: d["date"])`），排序後再截取 30 筆、計算 MA。

**如何提示使用者**：加入 warning「資料日期順序異常，已自動重新排序」，前端以黃色警告區塊顯示。

**決策原則**：MA 是基於時間順序的滑動窗口計算，排序錯誤會讓所有計算結果失效。自動修正比拒絕輸出更合理，因為排序是明確可修復的問題。

**實作狀態**：已實作。

## 4. 欄位缺失或為空

**如何偵測**：`data_processor.py` 有多層防護：
- 欄位數量檢查：`len(row) < 9` → 整筆視為異常
- 收盤價空值檢查：收盤價欄位為空字串 → 標記 `_bad_close`
- 收盤價 0 值檢查：原始字串非空但轉換後為 0.0 → 同樣標記 `_bad_close`（P0 新增）
- 非收盤價欄位空值檢查：開盤價、最高價、最低價的原始值為空或 `"--"` → 標記 `_bad_ohlc` 列表，記錄哪些欄位有問題（P1 新增）
- 數值轉換防護：`_to_float()` 處理 `""`、`"--"`、`"X"` 開頭等異常值，回傳 0.0
- 日期格式檢查：民國曆解析失敗 → 整筆略過

**是否繼續計算**：
- 整筆異常（欄位數不足、日期解析失敗）→ 略過該筆，加 warning
- 收盤價為空或異常為 0 → 略過該筆，加 warning（區分「空」與「異常為 0」兩種原因）
- 其他欄位為空（開盤價、最高價、最低價）→ 轉為 0.0 繼續處理（不略過），加 warning 提示圖表顯示可能異常（P1 新增）

**如何提示使用者**：warnings 列出每筆被略過或有問題的原因：
- 「資料異常，已略過一筆：{日期}」
- 「{日期} 收盤價空，該筆不納入計算」
- 「{日期} 收盤價異常為 0，該筆不納入計算」
- 「{日期} {欄位名}為空，圖表顯示可能異常」（P1 新增，欄位名如「開盤價」「最高價、最低價」）

**AI 分析防護**：`ai_analyzer.py` 的 `_fact_price_range()` 在取最高價時過濾 `high > 0`、取最低價時過濾 `low > 0`，避免產生「最低價為 0.00 元」的錯誤事實（P1 新增）。

**決策原則**：收盤價是核心計算欄位（MA5/MA20 都依賴它），缺了或異常為 0 都必須略過，避免污染均線計算。其他欄位缺失不影響核心計算，但需明確告知使用者顯示可能異常，且 AI 分析層需過濾 0 值避免產生錯誤事實。

**實作狀態**：已實作。收盤價防護完整（含空值與 0 值），非收盤價欄位空值有 warning 提示且 AI 分析已過濾 0 值。

## 5. 不同來源資料不一致

**如何偵測**：沒有做。目前系統設計上刻意迴避了這個問題。

**架構決策**：`config.py` 的 `DATA_SOURCE` 一次只選一個來源（seed 或 TWSE），歷史股價計算全程使用同一來源，不混用。`origin-spec.md` 明確記錄「統一使用 TWSE 傳統 API 單一來源，避免第三方轉手造成數值差異」。

**目前狀況**：TWSE OpenAPI / 本地 JSON 原本規劃作為補充來源（本益比、殖利率等 extra 資訊），但 extra 欄位尚未實作（`main.py` 中 `extra: None`），所以目前不存在跨來源比對的場景。

**未來風險**：如果要實作 extra 資訊，會需要處理 TWSE OpenAPI 日期格式（`1150402`）與 TWSE 傳統 API 日期格式（`115/04/02`）不同的問題。

**實作狀態**：刻意迴避（單一來源策略），未來擴充時需處理。

## 總結

| 情境 | 偵測 | 繼續/中止 | 提示使用者 | 實作狀態 |
|---|---|---|---|---|
| 連線失敗 | try/except + None 檢查 | 第一月失敗中止，後續月份失敗用已有資料繼續 | error → 紅色錯誤 | 已實作 |
| 筆數不足 | 清洗後計數 < 30/5/20 | 繼續，MA 不足處填 null | warnings → 黃色警告 | 已實作 |
| 日期排序錯誤 | 逐筆比較日期是否遞增 | 自動排序後繼續 | warnings → 黃色警告 | 已實作 |
| 欄位缺失/為空 | 欄位數 + 收盤價空值 + OHL 空值檢查 | 收盤價異常略過，OHL 空值繼續但加 warning | warnings 列出被略過與異常的筆 | 已實作 |
| 不同來源不一致 | — | — | — | 刻意迴避 |

### 背後決策原則

1. **寧可少顯示，不可誤顯示**：資料有疑慮時優先略過或標記，不會用有問題的數據算出看似正確的結果
2. **有多少就顯示多少**：不因部分缺失而完全不顯示，但一定告知使用者缺了什麼
3. **核心欄位嚴格、輔助欄位寬鬆**：收盤價（MA 計算依賴）缺失直接略過，其他欄位容許降級處理
4. **單一來源避免矛盾**：不混用不同資料來源的數值，從架構層面消除不一致的可能性

## (D) 驗證與 QA

### 1. 任選一天，驗證 MA5 計算（需說明資料與方法）

**選擇日期：2026-02-24（115/02/24），使用 `seed/stock_2330.json` 台積電資料**

**方法：** 根據 `data_processor.py:150-155` 的 `_sma()` 函式，MA5 = 最近 5 日收盤價的算術平均。原始資料共 36 筆，系統在日期排序驗證（line 38-46）後截取最後 30 筆（`cleaned[-30:]`），因此實際使用的是 index 6（02-10）起的資料。截取後 02-24 位於 truncated index=4，是 MA5 可計算的第一天。

**手動計算：**

| 日期 | 收盤價 | truncated index |
|------|--------|-----------------|
| 02-10 | 1,880.00 | 0 |
| 02-11 | 1,915.00 | 1 |
| 02-23 | 1,900.00 | 2 |
| 02-24 | 1,965.00 | 3 |
| 02-25 | 2,015.00 | 4 |

MA5 = (1880 + 1915 + 1900 + 1965 + 2015) / 5 = 9675 / 5 = **1935.00**

**驗證方式：** 程式碼取 `closes[0:5]`（即 truncated index 0~4），執行 `round(sum(window) / 5, 2)`，結果應為 1935.0。可透過啟動後端（`DATA_SOURCE = "seed"`）、呼叫 `/api/stock/2330` 並檢查回傳 data 中 date 為 `2026-02-25` 那筆的 `ma5` 欄位來驗證。

**補充說明：** 截取前系統會先進行日期排序驗證（line 38-46），確保資料時間順序正確後才截取與計算。若不考慮截取（例如資料筆數 ≤ 30），也可直接對原始 36 筆資料驗證。以原始 index=4 的 02-06 為例：MA5 = (1765 + 1800 + 1785 + 1765 + 1780) / 5 = 8895 / 5 = **1779.00**。

### 2. 說明你如何確認 MA20 正確；若資料不足 20 日，你如何處理？

**確認方式：**
- 同 MA5 驗證方法，取連續 20 日收盤價手動加總除以 20，與程式輸出比對。程式碼邏輯在 `data_processor.py:150-155`，`_sma(closes, i, 20)` 會取 `closes[i-19 : i+1]` 共 20 個值。
- 可選取最後一天（如 04-02）的 MA20 做交叉驗證：取該天往前數 20 個收盤價，手動計算 `sum / 20` 並 `round` 到小數第 2 位。

**資料不足 20 日的處理（實際程式碼行為）：**
1. `_sma()` 在 `index < period - 1`（即 index < 19）時回傳 `None`（line 152-153），前端會顯示為無資料點（圖表上該段 MA20 線不會繪出）。
2. `process_stock_data()` 在 `total < 20` 時加入警告訊息：`"資料不足 20 筆，無法計算 MA20"`（line 66-67）。
3. 例如 `stock_0050.json` 有 22 筆資料，前 19 天 MA20 為 `null`，僅最後 3 天有值。`stock_few.json` 僅 4 筆有效資料（1 筆收盤價為空被跳過），MA20 全部為 `null`。

**注意：** 系統並非不輸出，而是對應天數的 MA20 為 `null`，並附加警告。這是正確的處理方式——不補值、不外推。

### 3. 說明你如何檢查資料排序、缺漏值或異常值

**排序檢查：**
- `data_processor.py` 在清洗完成後、截取 30 筆之前，逐筆比較日期是否嚴格遞增（ISO 格式字串比較）。
- 若發現亂序 → 自動以日期排序，並加入 warning「資料日期順序異常，已自動重新排序」。
- `data_fetcher.py` 合併多月資料時用「舊月份放前面」維持時間順序，加上 processor 層的排序驗證，形成雙重保障。

**缺漏值檢查：**
- `_parse_row()`：欄位不足 9 個 → 回傳 `None`，該筆被略過並產生警告。
- 收盤價為空字串 → 標記 `_bad_close`，該筆不納入計算，並產生警告。
- 收盤價非空但轉換後為 0.0 → 同樣標記 `_bad_close`，產生「收盤價異常為 0」的警告（P0 新增）。
- 日期格式無法解析 → `_roc_to_ad()` 回傳 `None` → 該筆略過。
- `stock_few.json` 的第 3 筆（04/03）收盤價為空，實際測試可確認系統正確跳過並警告。

**異常值檢查：**
- `_to_float()` 處理了 `"X"` 前綴（TWSE 偶爾標記特殊值）、`"--"` 和空字串。
- 收盤價 0 值已有防護（P0 新增），不會參與 MA 計算。
- **仍未做數值範圍檢查**（例如收盤價是否為負、單日漲跌是否異常），列為 P2 待辦。

### 4. 指出 AI 分析中至少一個可能不嚴謹之處，並說明如何降低此問題

**不嚴謹之處：近期趨勢判斷邏輯過於簡化**

`ai_analyzer.py:155-171` 的 `_fact_recent_trend()` 判斷條件是：近 5 日中有 4 日以上收盤價上漲（`ups >= n-1`，即 `>= 4`）才輸出「呈上升趨勢」。

**問題：**
- 若 5 日中有 4 日微漲（每日漲 0.01 元）但 1 日暴跌 100 元，系統仍會輸出「呈上升趨勢」，但實際上淨變化是大幅下跌。**該函式只看方向、不看幅度。**
- 此外，AI 改寫模式（line 184-210）將事實交給 OpenAI API 改寫，prompt 雖然限制了「不可新增事實」，但 LLM 可能在措辭上隱含暗示（如「穩步上揚」暗示趨勢將延續），這無法透過 prompt 完全杜絕。

**降低方式：**
1. 趨勢判斷應納入**幅度考量**，例如計算近 5 日淨變動百分比，同時滿足方向一致且淨變動 > 某閾值才輸出趨勢描述。
2. AI 改寫的輸出應加入**後處理驗證**，例如檢查改寫結果是否包含原始 facts 中未出現的數字或比較級用語，若有則 fallback 到模板模式。

### 5. 在什麼情況下，你會選擇「不輸出 AI 分析」？

根據 `ai_analyzer.py:7-30` 的 `generate_analysis()` 實際邏輯，以下情況不輸出（回傳 `INSUFFICIENT_DATA` 或 `None`）：

1. **事實不足 2 項**（line 17-19）：`extract_facts()` 提取的事實少於 2 條時，回傳 `"依據不足，無法產生分析"`。例如僅有 1 天有效資料，只能提取「價格範圍」一項事實。

2. **無有效資料**（line 35-36）：`data` 為空列表時，`extract_facts()` 直接回傳空 list，觸發上述條件。

3. **從更上層來看**（`main.py` 中），若 `process_stock_data()` 回傳 `error` 欄位或 `data` 為空，`generate_analysis()` 根本不會被呼叫，analysis 欄位為 `None`。

**作為 QA，建議額外在以下情況也不輸出分析：**
- 資料天數過少（如不足 10 天），即使能提取 2 項事實，分析的代表性也很低。
- 存在大量被跳過的異常資料筆數（例如 30 筆中有 10 筆被跳過），此時剩餘資料的連續性和代表性存疑。
- ~~收盤價出現 0 值~~ → 已修復（P0：非空但為 0 的收盤價會被略過並產生警告）。

  - 做法：逐筆比較 `date` 欄位是否嚴格遞增，若發現亂序則主動以日期排序，並加入 warning `"資料日期順序異常，已自動重新排序"`
  - 原因：MA 是基於「時間順序」的滑動窗口計算，排序錯誤會讓所有 MA 值與趨勢判斷全部失效，是整個系統計算正確性的前提

- [x] **收盤價 0 值防護**
  - 位置：`data_processor.py` 的 `_parse_row()` 或清洗迴圈中
  - 做法：收盤價經 `_to_float()` 轉換後若為 0.0（且原始字串非空），視同異常，標記 `_empty_close = True` 並加入 warning `"{date} 收盤價異常為 0，該筆不納入計算"`
  - 原因：0 值混入 closes 列表會將 MA 均線拉低數十甚至數百元，產出的圖表和分析完全失真，且使用者無從察覺

#### P1 — 應盡快實作（顯示品質與分析可信度）

- [x] **非收盤價欄位（開盤、最高、最低）空值 warning**
  - 位置：`data_processor.py` 的 `_parse_row()` + 清洗迴圈，`ai_analyzer.py` 的 `_fact_price_range()`
  - 做法：`_parse_row()` 檢查 open/high/low 原始值是否為空或 `"--"`，記錄到 `_bad_ohlc` 列表；清洗迴圈中讀取該列表加入 warning `"{date} {欄位名}為空，圖表顯示可能異常"`（不略過該筆）；`_fact_price_range()` 額外過濾 `high > 0` 和 `low > 0`，避免產生「最低價為 0.00 元」的錯誤事實
  - 測試資料：`seed/stock_ohlc_empty.json`（含開盤價空、最高/最低價 `"--"`、三欄全空三種情境）
  - 原因：`_fact_price_range()` 用 high/low 來產生「期間最高/最低價」事實，若 low 為 0 會輸出「最低價為 0.00 元」，直接破壞 AI 分析的可信度

- [x] **趨勢判斷納入幅度考量**
  - 位置：`ai_analyzer.py` 的 `_fact_recent_trend()`
  - 做法：在現有方向判斷（`ups >= n-1`）之後，額外計算 `net_change = (closes[-1] - closes[0]) / closes[0]`，僅在方向次數達標**且**淨變動方向一致時才輸出趨勢描述（如 `ups >= 4 and net_change > 0`）
  - 原因：「4 日微漲 + 1 日暴跌」仍被判定為上升趨勢，會讓使用者對走勢產生錯誤認知，違反系統「不可誤顯示」的設計原則

- [x] **AI 分析輸出門檻提高**
  - 位置：`data_processor.py`（新增 `raw_count`）、`ai_analyzer.py` 的 `generate_analysis()`、`main.py`（傳入 `raw_count`）
  - 做法：`data_processor.py` 回傳 `raw_count`（原始資料筆數）；`generate_analysis()` 新增參數 `raw_count`，在事實提取前加兩個門檻：`len(data) < 10` → 拒絕輸出、`(raw_count - len(data)) / raw_count > 0.3` → 拒絕輸出
  - 原因：2-3 天資料雖然能湊出 2 項事實，但「期間價格變化」只反映 1-2 天波動，不具代表性；大量略過則代表資料連續性被破壞，分析基礎不可靠

#### P2 — 建議實作（防禦性增強）

- [x] **AI 改寫後處理驗證**
  - 位置：`ai_analyzer.py` 新增 `_validate_rewrite()` 和 `_extract_numbers()`；`analysis_templates.py` 新增 `BANNED_KEYWORDS`
  - 做法：`_ai_rewrite()` 取得 LLM 輸出後呼叫 `_validate_rewrite(text, facts)` 做兩道驗證：(1) 禁用詞檢查 — 掃描 20 個投資建議/預測相關詞彙，命中即不通過；(2) 數字比對 — 提取 facts 和輸出中的所有數字（統一去逗號去尾零），輸出包含 facts 中不存在的數字即不通過。驗證失敗回傳 `None`，觸發既有的 fallback 到模板模式，不做重試
  - 原因：prompt 約束不等於行為保證，LLM 有機率違反指令，加入程式層面的防線可降低輸出不當內容的風險

- [x] **數值合理性範圍檢查**
  - 位置：`data_processor.py` 的 `_parse_row()`（負數檢查）+ `process_stock_data()`（漲跌幅檢查，置於排序驗證之後、截取 30 筆之前）
  - 做法：收盤價為負數 → 標記 `_bad_close`（reason「異常為負數」），略過該筆；單日漲跌幅超過 ±50%（相對前一日）→ 加入 warning `"{date} 價格波動異常（{change}%），請確認資料正確性"`（不略過，因為無法區分極端行情與解析錯誤）
  - 原因：解析錯誤（如逗號位置錯位）可能產生不合理的價格值，目前系統會照單全收，加入範圍檢查可作為最後一道防線

- [x] **跨來源日期格式統一工具**
  - 位置：`data_processor.py` 的 `_roc_to_ad()`
  - 做法：擴充既有函式，新增 `elif` 分支處理 7 位純數字格式（`s.isdigit() and len(s) == 7`），取前 3 碼為民國年、中間 2 碼月、末 2 碼日。兩種格式以有無 `/` 區分，不衝突。不另建 `date_utils.py`（目前只有一個函式用到，不需要獨立模組）
  - 原因：為未來實作 extra 欄位（本益比、殖利率）預做準備，避免屆時才處理格式衝突造成資料錯配
